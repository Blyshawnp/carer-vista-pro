# Packaging And Sharing Guide

This app is a Next.js + Supabase PWA. That means there are three different ways to "package" it:

- share the source code with another person so they can run their own copy
- package the hosted PWA as an Android APK/AAB
- package it for Google Play or Apple's App Store

Do not share `.env.local`, Supabase service role keys, signing keys, or store credentials.

## 1. Share The App With Another Person

Use this when another person wants their own separate copy of the app.

### What To Send

Send the project folder without generated files or secrets.

Do include:

- `src`
- `public`
- `supabase`
- `docs`
- `package.json`
- `package-lock.json`
- `README.md`
- `.env.local.example`
- config files such as `tsconfig.json`, `tailwind.config.ts`, `next.config.mjs`, `postcss.config.mjs`, `vercel.json`

Do not include:

- `.env.local`
- `node_modules`
- `.next`
- `.git`, unless you intentionally want to share the Git history
- Android signing keys, Apple certificates, provisioning profiles, or app store credentials

### Create A Zip On Windows

From the folder above this app:

```powershell
Compress-Archive `
  -Path caregiver-app `
  -DestinationPath caregiver-app-source.zip `
  -Force
```

Before sending the zip, open it and remove these if they are present:

- `caregiver-app/.env.local`
- `caregiver-app/node_modules`
- `caregiver-app/.next`
- `caregiver-app/.git`

### Setup Steps For The Other Person

They need:

- Node.js 20 or newer
- a Supabase project
- optionally, a Vercel account for hosting

Setup:

```powershell
npm install
Copy-Item .env.local.example .env.local
npm run dev
```

Then they fill `.env.local` with their own Supabase values:

```env
NEXT_PUBLIC_SUPABASE_URL=
NEXT_PUBLIC_SUPABASE_ANON_KEY=
SUPABASE_SERVICE_ROLE_KEY=
NEXT_PUBLIC_VAPID_PUBLIC_KEY=
VAPID_PRIVATE_KEY=
VAPID_SUBJECT=
```

Important: this repository has migration files in `supabase/migrations`, but confirm the full initial database schema is present before another person creates a fresh Supabase project. If the fresh project is missing base tables like `profiles`, `organizations`, `clients`, `shifts`, `check_ins`, `todo_templates`, or `shift_todos`, export the schema from the current Supabase project or add a complete initial migration before sharing.

## 2. Prepare The PWA For Mobile Packaging

Before Android or iOS packaging, deploy the web app to a stable HTTPS URL.

Recommended hosting:

- Vercel for the Next.js app
- Supabase for database, auth, storage, edge functions, and scheduled jobs

Checklist:

- `npm run build` passes
- the deployed app works over `https://`
- login works
- `/manifest.json` loads publicly
- `/sw.js` loads publicly
- icons exist in `public`
- Supabase Auth redirect URLs include the production domain
- all required environment variables are configured on the host
- test accounts exist for app review
- no test data, placeholder text, or fake users remain unless they are intentional demo accounts

## 3. Android APK For Direct Sharing

Use this when you want a file someone can install manually on Android.

The simplest route is a Trusted Web Activity generated by Bubblewrap. Bubblewrap wraps the hosted PWA in a small Android shell.

Requirements:

- deployed HTTPS app URL
- Node.js
- Java/JDK and Android SDK, or let Bubblewrap install/configure what it needs
- Android device or emulator for testing

Install Bubblewrap:

```powershell
npm install -g @bubblewrap/cli
```

Create a separate wrapper folder outside the Next.js app:

```powershell
mkdir caregiver-android
cd caregiver-android
bubblewrap init --manifest=https://YOUR-DOMAIN.com/manifest.json
bubblewrap build
```

The build creates Android release files. Depending on the Bubblewrap version/configuration, this commonly includes:

- `app-release-signed.apk` for direct Android install/testing
- `app-release-bundle.aab` for Google Play

Install on a connected Android device:

```powershell
bubblewrap install
```

If Trusted Web Activity verification fails, Android may show browser UI. Fix this by adding the generated Digital Asset Links file to the hosted app at:

```text
https://YOUR-DOMAIN.com/.well-known/assetlinks.json
```

In this Next.js project, that means adding:

```text
public/.well-known/assetlinks.json
```

Then redeploy the website and rebuild/retest the Android package.

Keep the Android signing key safe. Losing it can block future app updates.

## 4. Google Play Upload

Google Play generally expects an Android App Bundle, not just an APK, for new production apps.

Use the Bubblewrap-generated:

```text
app-release-bundle.aab
```

Google Play Console checklist:

- create a Google Play Developer account
- create the app in Play Console
- complete app name, description, screenshots, icon, feature graphic, category, and contact details
- complete Data Safety and privacy forms
- provide a privacy policy URL
- set content rating
- configure countries/regions
- upload the `.aab` to Internal testing first
- test install from Play's internal testing track
- promote to Closed testing, Open testing, or Production when ready

For this app, prepare review/testing credentials. Because the app requires login, reviewers need a working demo account and enough demo data to verify scheduling, check-in/check-out, tasks, messages, and notifications.

Official references:

- Trusted Web Activities and Bubblewrap: https://developer.chrome.com/docs/android/trusted-web-activity/quick-start/
- Google Play releases: https://support.google.com/googleplay/android-developer/answer/9859348

## 5. Apple And iOS Options

There are two separate iPhone paths:

- PWA install from Safari: users open the website in Safari and choose Add to Home Screen.
- App Store app: you need a native iOS wrapper/project and App Store review.

You cannot upload an APK to Apple. Apple requires an iOS app archive uploaded through Xcode, Transporter, or App Store Connect tooling.

### Option A: Safari PWA

This is the simplest iPhone path.

Steps:

- deploy the app to HTTPS
- open it in Safari on iPhone
- tap Share
- choose Add to Home Screen

This does not require Apple Developer Program membership, but it also does not appear in the App Store.

### Option B: App Store Wrapper

Use this when you want the app listed in Apple's App Store.

Recommended wrapper approach:

- Capacitor
- Xcode
- Apple Developer Program membership

High-level steps:

```powershell
npm install @capacitor/core @capacitor/cli
npx cap init "Caregiver" "com.yourcompany.caregiver"
npm install @capacitor/ios
npx cap add ios
```

Because this is a server-rendered Next.js app, the iOS wrapper should usually point to the deployed production URL instead of trying to bundle the whole Next app as static files. Configure Capacitor's server URL for the hosted app, then open the iOS project:

```powershell
npx cap open ios
```

In Xcode:

- set the Bundle Identifier
- configure signing with an Apple Developer account
- set app icon and launch screen
- verify camera/location/notification permission descriptions if used
- test on a real iPhone
- archive the app
- upload the archive to App Store Connect

Apple review checklist:

- provide a privacy policy URL
- provide demo login credentials
- make sure the backend is live during review
- remove test or placeholder content
- explain any background location, notifications, or sensitive caregiving/health-related behavior in review notes
- make the app feel complete enough that it is not rejected as only a thin website wrapper

Official references:

- Apple App Review Guidelines: https://developer.apple.com/app-store/review/guidelines/
- Upload builds to App Store Connect: https://developer.apple.com/help/app-store-connect/manage-builds/upload-builds
- Xcode upload flow: https://help.apple.com/xcode/mac/current/en.lproj/dev442d7f2ca.html

## 6. Release Checklist

Before sending to anyone or uploading to stores:

- run `npm run build`
- test login, schedule, shift creation, check-in, check-out, tasks, clients, team, messages, notifications, payroll, and incidents
- verify Supabase RLS policies with admin, caregiver, family, and client users
- clean test data using `CLEANUP_TEST_DATA.md`
- create intentional demo data only if needed for reviewers
- rotate any keys that were accidentally shared
- verify production environment variables
- verify Supabase Auth redirect URLs
- verify PWA manifest and icons
- verify Android Digital Asset Links
- keep signing keys backed up securely
- store final `.apk`, `.aab`, and iOS archive/export files outside the app source folder
